Trabajo práctico

Para aprobar la materia se debe presentar un trabajo práctico, cuyo tema será de elección del alumno o grupo de alumnos que realizarán el trabajo, dando preferencia al desarrollo en Groovy y Grails de un sistema de gestión que integre tecnologías y conceptos innovadores (por ejemplo Web 2.0). El alcance del trabajo se definirá en función de la cantidad de participantes, pudiendo variar desde un trabajo práctico individual hasta un grupo de a lo sumo cuatro participantes.

Entrega parcial

Se debe realizar una entrega previa durante la cursada de la materia que consiste en:

  • Un texto breve que describe en grandes rasgos el dominio sobre el cuál se trabaja
  • Una serie de historias de usuario -cuyo número dependerá de la cantidad de integrantes del grupo-
  • Un modelo de dominio que dé soporte a las historias escritas
  • Un modelo de dominio implementado con sus test unitarios correspondientes demostrando que el dominio no es anémico.

Para esta entrega deberá incluir el código fuente del modelo. Se espera que sea un link a un archivo descargable que contenga todo el código fuente o un link a un repositorio, con preferencia de este último, en especial si cuenta con sistema para cargar issues como tiene GitHub, Assembla y otros.

El modelo de dominio debe estar implementado en alguno de los lenguajes que corren sobre la JVM (Groovy, Scala, Clojure, Java, etc.) usando los conceptos de orientación a objetos aprendidos en Algoritmos III y los explicados en las clases sobre Domain-Driven Design, de forma tal que a partir de la lectura del código sean evidentes los conceptos y funcionalidades involucrados. Técnicamente, el dominio del problema debe estar modelado con lo que se conoce como “POJOs”.

Entrega final

La entrega final consiste en la presentación de una aplicación en donde se implementen las historias de usuario. Para esto es necesario integrar una serie de tecnologías y frameworks que brinden la infraestructura adecuada al modelo del dominio. Se sugiere la utilización del Framework Grails para el caso de usar Groovy o de los Frameworks Spring para resolver la inyección de dependencias e Hibernate para resolver la persistencia.

El trabajo práctico se presentará en alguna fecha de coloquio habilitada. Para su aprobación, además de las preguntas acerca de lo realizado en el trabajo práctico, se podrán hacer preguntas sobre los temas conceptuales explicados en clase, especialmente los que tiene que ver con “Domain Driven Design”. En el trabajo práctico se prestará especial atención a que el modelo de dominio NO tenga las características de un “modelo de dominio anémico” (concepto que se explicará oportunamente). Es muy importante llevar las historias de usuario entregadas y aprobadas en la cursada para rendir el coloquio, llevando a la desaprobación en caso de no contar con ellas dado que se utilizarán para la corrección.

Para realizar esta entrega se deberá tener aprobada la cursada para lo cual se deberá:

  • Tener el 75% de la asistencia a clases.
  • Tener aprobado la entrega parcial.

En caso de no cumplir con los requisitos no se podrá rendir como alumno regular, con lo que se deberá contactar a los docentes.
